# user_stories_v1.md

# Ardtire Society Civic Digital Governance Platform
## User Stories v1

## Document Status

- Version: v1
- Purpose: comprehensive recommended baseline user-story inventory derived from product intent
- Scope: platform product stories, not implementation tasks
- Note: this is intended to be broad and practically comprehensive for v1/MVP-plus planning, not a claim that no future stories will ever exist

---

# 1. Story Format

Each story is written in the form:

**As a [user type], I want [capability], so that [outcome].**

Where useful, stories are grouped by domain and workflow.

---

# 2. User Roles Covered

- Public visitor
- Registered user
- Membership applicant
- Approved member
- Moderator
- Membership reviewer
- Editor / publisher
- Administrator
- System operator / auditor
- Future governance officer or clerk

# 3. Public Site and Discovery Stories

## 3.1 Public understanding

1. **As a public visitor, I want to read a clear explanation of the Ardtire Society, so that I can understand what the organization is and why it exists.**

2. **As a public visitor, I want to understand the purpose of the Civic Digital Governance Platform, so that I know what the platform is for.**

3. **As a public visitor, I want to see the pathway from visitor to registered user to member, so that I understand how participation works.**

4. **As a public visitor, I want to read public-facing constitutional and governance information, so that I can understand the seriousness and structure of the Society.**

5. **As a public visitor, I want to see public updates, announcements, or published materials, so that I can follow the Society’s development.**

6. **As a public visitor, I want to view public governance documents and official statements, so that I can assess the institution’s credibility.**

7. **As a public visitor, I want the public site to feel serious, trustworthy, and coherent, so that I feel confident the Society is building something real and organized.**

## 3.2 Public calls to action

8. **As a public visitor, I want clear calls to register or learn more, so that I know what action to take next.**

9. **As a public visitor, I want to know whether I must become a member before participating in governance, so that I understand the rules of access.**

10. **As a public visitor, I want to know what eligibility or expectations apply to membership, so that I can decide whether to apply.**

---

# 4. Registration, Authentication, and Account Stories

## 4.1 Account creation

11. **As a public visitor, I want to register an account with a username, email, and password, so that I can access authenticated platform features.**

12. **As a public visitor, I want to verify my email address, so that my account can be trusted and secured.**

13. **As a public visitor, I want to be informed if an email or username is already in use, so that I can resolve registration issues quickly.**

14. **As a public visitor, I want registration errors explained clearly, so that I know how to fix them.**

## 4.2 Authentication

15. **As a registered user, I want to sign in securely, so that I can access my account and platform features.**

16. **As a registered user, I want to sign out securely, so that I can protect my account on shared devices.**

17. **As a registered user, I want to reset my password if I forget it, so that I can regain access.**

18. **As a registered user, I want to change my password, so that I can maintain my account security.**

19. **As a registered user, I want to manage active sessions or devices, so that I can revoke access if needed.**

20. **As a registered user, I want to use strong authentication protections, so that my account is harder to compromise.**

## 4.3 Account status awareness

21. **As a registered user, I want to see whether I am merely registered, currently applying, approved as a member, suspended, or otherwise restricted, so that I understand my current standing.**

22. **As a registered user, I want to know which platform actions I am permitted to take, so that I do not waste time attempting unavailable actions.**

23. **As a registered user, I want to see what next steps are available to me, so that I can progress toward deeper participation.**

# 5. Profile and Identity Stories

24. **As a registered user, I want a profile page, so that I can review my account details and participation status.**

25. **As a registered user, I want to edit my basic profile information, so that my account reflects accurate information about me.**

26. **As a registered user, I want to control what profile information is visible to others, so that I can balance participation with privacy.**

27. **As a member, I want my identity on proposals, comments, and votes to be handled according to platform rules, so that accountability and privacy are managed correctly.**

28. **As an administrator, I want identity records tied reliably to user actions, so that the platform preserves accountability.**

# 6. Membership Application Stories

## 6.1 Starting an application

29. **As a registered user, I want to begin a membership application, so that I can seek Society membership.**

30. **As a registered user, I want the platform to explain the membership process before I begin, so that I know what will happen.**

31. **As a registered user, I want to know what information is required in the application, so that I can prepare it properly.**

32. **As a registered user, I want to save a draft application, so that I can complete it over time.**

33. **As a registered user, I want to resume an in-progress application, so that I do not lose my work.**

## 6.2 Submitting an application

34. **As a registered user, I want to submit my completed membership application, so that it enters the review process.**

35. **As a registered user, I want validation feedback before submission, so that I can correct missing or invalid information.**

36. **As a registered user, I want a confirmation when my application is submitted, so that I know it was received.**

37. **As a registered user, I want to see my application status after submission, so that I know where it stands.**

## 6.3 Application status and outcomes

38. **As an applicant, I want to know whether my application is pending, under review, approved, rejected, returned for more information, or withdrawn, so that I understand my status.**

39. **As an applicant, I want to be notified when my application status changes, so that I can respond appropriately.**

40. **As an applicant, I want to supply additional information if requested, so that I can continue the review process.**

41. **As an applicant, I want to withdraw my application before decision when permitted, so that I can stop the process if I choose.**

42. **As an approved applicant, I want my account status to update automatically to member, so that I gain the correct permissions without confusion.**

43. **As a rejected applicant, I want to know whether I may reapply later, so that I understand my future options.**

# 7. Membership Review and Administration Stories

## 7.1 Reviewer workflows

44. **As a membership reviewer, I want a queue of submitted applications, so that I can process them efficiently.**

45. **As a membership reviewer, I want to filter applications by status, age, or priority, so that I can work systematically.**

46. **As a membership reviewer, I want to open an application and view all submitted information, so that I can evaluate it properly.**

47. **As a membership reviewer, I want to request additional information from an applicant, so that incomplete applications can proceed.**

48. **As a membership reviewer, I want to approve an application, so that a qualified user becomes a member.**

49. **As a membership reviewer, I want to reject an application with a reason, so that the outcome is recorded clearly.**

50. **As a membership reviewer, I want to return an application for revision or clarification, so that applicants can fix issues without restarting entirely.**

51. **As a membership reviewer, I want my decision actions recorded in an audit trail, so that the process remains accountable.**

## 7.2 Membership status management

52. **As an administrator, I want to view current members and their statuses, so that I can manage participation eligibility.**

53. **As an administrator, I want to change a member’s status when necessary, so that the platform reflects governance reality.**

54. **As an administrator, I want to suspend or restrict a member where permitted by policy, so that the platform can enforce order and safety.**

55. **As an administrator, I want status changes to affect permissions immediately, so that access control remains accurate.**

56. **As an administrator, I want historical membership state changes preserved, so that institutional records remain intact.**

# 9. Proposal Submission Stories

## 9.1 Creating a proposal

62. **As a member, I want to create a new constitutional or governance proposal, so that I can contribute to institutional development.**

63. **As a member, I want proposal templates or structured fields, so that submissions are consistent and intelligible.**

64. **As a member, I want to categorize my proposal, so that it can be organized and reviewed appropriately.**

65. **As a member, I want to provide a title, summary, rationale, and proposed text, so that others can understand and assess my submission.**

66. **As a member, I want to save proposal drafts, so that I can work on them before formal submission.**

67. **As a member, I want to edit my draft proposal before submission, so that I can refine it.**

68. **As a member, I want validation feedback before submission, so that I know my proposal is complete enough to enter process.**

## 9.2 Submitting a proposal

69. **As a member, I want to submit my proposal into the governance workflow, so that it becomes formally reviewable.**

70. **As a member, I want a confirmation when I submit a proposal, so that I know it entered the system correctly.**

71. **As a member, I want the initial status of my proposal to be visible immediately, so that I know what stage it is in.**

72. **As a member, I want to withdraw or retract a proposal draft or early-stage proposal when permitted, so that I retain appropriate control over my submission.**

# 10. Proposal Review and Workflow Stories

## 10.1 State model visibility

73. **As a member, I want every proposal to display its current workflow state, so that I understand where it is in the process.**

74. **As a member, I want the platform to show allowed next steps for a proposal, so that procedural transitions are understandable.**

75. **As an administrator or moderator, I want proposal states to be explicit and controlled, so that the workflow remains legitimate and consistent.**

## 10.2 State transitions

76. **As an authorized operator, I want to move a proposal between permitted workflow states, so that the process can advance.**

77. **As an authorized operator, I want forbidden state transitions blocked by the platform, so that procedural integrity is preserved.**

78. **As a member, I want to know why a proposal changed state, so that the process remains transparent.**

79. **As a member, I want timestamps and responsible actors recorded for state changes, so that workflow history is auditable.**

## 10.3 Review actions

80. **As a moderator or reviewer, I want to return a proposal for revision, so that weak or incomplete proposals can improve.**

81. **As a moderator or reviewer, I want to mark a proposal ready for discussion or review, so that members know it can be engaged.**

82. **As a moderator or reviewer, I want to mark a proposal eligible for voting when it satisfies requirements, so that voting only occurs at the right stage.**

83. **As a moderator or reviewer, I want to reject or close a proposal where policy requires it, so that invalid items do not linger ambiguously.**

84. **As a moderator or reviewer, I want to supersede a proposal with a newer or merged one when appropriate, so that the record reflects reality.**

# 12. Drafting and Version History Stories

## 12.1 Proposal revisions

90. **As a member, I want proposals to support multiple revisions, so that drafting can evolve through collaboration.**

91. **As a member, I want each revision to be preserved rather than overwritten, so that the drafting record remains durable.**

92. **As a member, I want to compare one revision to another, so that I can see exactly what changed.**

93. **As a member, I want revision notes or rationale, so that I know why changes were made.**

94. **As a moderator or reviewer, I want to reference a specific revision when requesting changes or approving movement, so that process remains precise.**

## 12.2 Authoritative text

95. **As a member, I want to know which text version is currently authoritative, so that I do not confuse a proposal draft with official draft language.**

96. **As a public reader or member, I want adopted or current constitutional text to be clearly distinguished from proposals and drafts, so that authority is not ambiguous.**

97. **As an editor or administrator, I want authoritative text updates to be controlled and traceable, so that publication remains legitimate.**

# 28. Recommended Core MVP Story Set

If a strict MVP must be defined, the highest-priority stories are these:

1. public visitor can understand the Society and participation pathway;
2. visitor can register an account;
3. registered user can log in securely;
4. registered user can start and submit a membership application;
5. reviewer can review and decide applications;
6. approved applicant becomes a member with upgraded permissions;
7. member can create and submit a proposal;
8. member can read and discuss proposals;
9. proposal revisions are preserved;
10. proposal workflow state is visible;
11. authorized operator can move proposal through defined states;
12. eligible member can vote during an active voting window;
13. outcome is recorded and visible;
14. authoritative drafting state is published distinctly from proposals;
15. administrators and moderators have basic operational tools;
16. important actions are auditable.

---

# 29. Product Story Summary

The platform should support a complete civic progression:

- discover,
- register,
- apply,
- be reviewed,
- become a member,
- submit proposals,
- deliberate,
- revise,
- vote,
- record outcomes,
- publish authoritative text,
- and preserve institutional history.

That progression is the heart of the product.
